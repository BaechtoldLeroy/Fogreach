# WP06 — Loot + Item-Namen Migration

## Branch Strategy

- Planning/base branch: `main`
- Stack base: `WP01` (use `--base WP01`)
- Merge target: `main`

## Objective

Migriere `loot.js` und `lootSystem.js` (~10 Strings: Quest-Item-Namen, Loot-Drop-Toasts, Gold-Hinweise) auf i18n-Keys.

## Context

- `loot.js` hält Item-Definitionen inkl. Quest-Item-Namen.
- `lootSystem.js` orchestriert Drops und zeigt Toasts (Gold gewonnen, Item aufgenommen, etc.).
- WP01 garantiert `window.i18n`.

## Implementation

### Subtask T027 — `loot.js` Strings registrieren + migrieren

**Steps**:
1. Modulkopf von `js/loot.js`:
   ```js
   if (window.i18n) {
     window.i18n.register('de', {
       'loot.item.cellar_key': 'Kellerschlüssel',
       'loot.item.archive_fragment': 'Archiv-Fragment',
       // ...alle Quest-Items + Loot-Konstanten
     });
     window.i18n.register('en', {
       'loot.item.cellar_key': 'Cellar Key',
       'loot.item.archive_fragment': 'Archive Fragment',
     });
   }
   ```
2. Item-Konstanten/Definitionen: statt hardcoded `name: 'Kellerschlüssel'` entweder `nameKey: 'loot.item.cellar_key'` und an Render-Site `i18n.t(item.nameKey)` aufrufen, ODER getter-pattern: `get name() { return i18n.t('loot.item.cellar_key'); }`.
3. **Konsistenz**: Konvention für alle Item-Definitionen: `nameKey`-Field, kein `name` mehr. Render-Sites (Tooltips, Inventar) müssen `i18n.t(item.nameKey)` aufrufen — das berührt potenziell `inventory.js` etc., aber die sind out-of-scope. → Falls Tooltip-Code nicht in diesem WP gehört: Item bekommt sowohl `nameKey` als auch `name` (rechnet sich aus Key zur Render-Zeit) als getter.

**Files**: `js/loot.js` (modify)

### Subtask T028 — `lootSystem.js` Strings registrieren + migrieren

**Steps**:
1. Modulkopf von `js/lootSystem.js`:
   ```js
   if (window.i18n) {
     window.i18n.register('de', {
       'loot.toast.gold_gained': '+{amount} Gold',
       'loot.toast.item_picked': '{item} aufgenommen',
       'loot.toast.inventory_full': 'Inventar voll!',

     });
     window.i18n.register('en', {
       'loot.toast.gold_gained': '+{amount} gold',
       'loot.toast.item_picked': 'Picked up {item}',
       'loot.toast.inventory_full': 'Inventory full!',
     });
   }
   ```
2. Toast-Spawn-Code auf `i18n.t('loot.toast.xxx', {amount: 50, item: i18n.t(item.nameKey)})`.

**Files**: `js/lootSystem.js` (modify)

### Subtask T029 — Item-Name-Auflösung an Render-Sites

**Steps**:
1. Wo in `loot.js`/`lootSystem.js` ein Item-`name` gerendert wird: `i18n.t(item.nameKey)`.
2. Für externe Verbraucher (z.B. `inventory.js` Tooltip): Helper exposen `window.LootSystem.getItemName(item) → string`, der intern `i18n.t(item.nameKey)` aufruft. So müssen externe Caller nichts ändern.
3. Falls Items in localStorage gespeichert werden (Save-Game): nur `nameKey` speichern, Name dynamisch auflösen — sonst gespeicherte Items hätten alte Sprache.

**Files**: `js/loot.js`, `js/lootSystem.js` (modify)

## Risks

- **Save-Game-Migration**: alte Saves haben `name: 'Kellerschlüssel'`. → Mitigation: beim Laden Fallback `item.nameKey || legacyNameLookup(item.name)`. Oder einfach: Items im Save haben sowieso ihre `id` — Name wird aus globalem ITEM_REGISTRY rekonstruiert. Beibehalten.
- **Cross-File-Konsumenten** (z.B. `inventory.js` Tooltip): wenn die `item.name` lesen statt `i18n.t(item.nameKey)`, sehen sie `undefined`. → Mitigation: getter-Pattern auf Item: `get name() { return window.i18n ? i18n.t(this.nameKey) : '???'; }` — externe Verbraucher merken nichts.

## Reviewer Guidance

- Save-Game laden + Sprachwechsel → Item-Namen aktualisieren.
- Quest-Item-Reward (z.B. Quest-Abgabe) zeigt korrekten Namen.
