# WP03 — Desktop parity + layout polish

## Objective

Ensure the mobile skill selection behavior is consistent with desktop, the overlay does not obstruct gameplay, and the overall layout is polished for release.

## Context

- WP01 adds the core selection overlay and swap logic.
- WP02 adds cooldown indicators.
- This WP handles edge cases, consistency with desktop behavior, and layout refinements.
- Desktop loadout system lives in `js/inventory.js` and uses keyboard shortcuts.

## Subtasks

### T001 — Sync equipped skills with desktop loadout state

**Purpose**: Ensure skill changes made on mobile are reflected if the player switches to desktop, and vice versa.

**Steps**:
1. Verify that `abilitySystem.js` is the single source of truth for equipped skills (both mobile and desktop read/write here).
2. If `inventory.js` maintains a separate equipped state, unify it with `abilitySystem.js`.
3. On scene restart or device change, mobile buttons re-read from the canonical equipped state.

**Files**: `js/abilitySystem.js`, `js/inventory.js`, `js/mobileControls.js`.

**Validation**: Equipping a skill on mobile and refreshing on desktop (or vice versa) shows the same loadout.

### T002 — Overlay layout does not obstruct gameplay area

**Purpose**: Ensure the skill selection overlay is properly positioned and dismissible.

**Steps**:
1. Center the overlay with a maximum width/height that leaves visible game area around the edges.
2. Add a semi-transparent backdrop that blocks game input while overlay is open.
3. Pause or reduce game activity while overlay is open (prevent death while selecting skills).
4. Ensure overlay works in both portrait and landscape orientations.
5. Respect safe-area insets from `mobileSafeArea.js` (if available from spec 021).

**Files**: `js/mobileControls.js`, `js/mobileAbilityButtons.js`.

**Validation**: Overlay is usable in both orientations, does not clip off-screen, and game input is blocked while open.

### T003 — Edge cases and polish

**Purpose**: Handle edge cases for a smooth player experience.

**Steps**:
1. If a skill is on cooldown when swapped out, the new skill starts with no cooldown.
2. If the player is mid-animation when opening the overlay, queue the overlay until the animation completes (or open immediately if safe).
3. Add a brief transition animation (fade in/out) for the overlay.
4. Ensure the swap button/trigger does not appear on desktop (mobile-only).

**Files**: `js/mobileControls.js`, `js/mobileAbilityButtons.js`, `js/abilitySystem.js`.

**Validation**: Swapping skills during combat edge cases do not cause errors. Overlay transitions are smooth.

## Definition of Done

- Mobile skill selection is functionally equivalent to desktop loadout.
- Overlay respects safe areas and works in both orientations.
- Game input is blocked while overlay is open.
- Skill swap edge cases (cooldowns, mid-animation) are handled gracefully.
- Swap trigger is mobile-only; desktop is unaffected.
- No regressions in desktop gameplay.

## Risks

- Pausing game activity during overlay may not be possible if the game uses a single update loop. Alternative: make the player invulnerable while overlay is open.
- Unifying desktop and mobile equipped state could surface bugs if `inventory.js` and `abilitySystem.js` have diverged.
