# WP04 — Rebalance event weights + anti-repetition + depth scaling

## Objective

Rebalance the event weight distribution now that the pool has expanded from 6 to 12 event types. Reduce chest-event dominance, add anti-repetition logic to prevent the same event firing twice in a row, and tune depth-based probability scaling.

## Context

After WP02 and WP03, the full event pool is:
- `treasure_cache` (currently 30) — reduce to ~15
- `ambush` (currently 18) — reduce to ~12
- `wandering_merchant` (currently 18) — keep at ~15
- `trapped_chest` (currently 12) — reduce to ~8
- `lore_fragment` (currently 15) — keep at ~12
- `environmental_hazard` (currently 7) — keep at ~7
- `shrine_buff` (new, ~10)
- `npc_rescue` (new, ~8)
- `cursed_item` (new, ~6)
- `elite_ambush` (new, ~8)
- `puzzle_room` (new, ~7)
- `gambling` (new, ~6)

Total weight: ~114. Chest events (treasure_cache + trapped_chest) drop from 42% to ~20%.

## Subtasks

### T001 — Rebalance existing event weights

**Purpose**: Reduce chest-event dominance and balance the expanded pool.

**Steps**:
1. Update `treasure_cache` weight from 30 to 15.
2. Update `ambush` weight from 18 to 12.
3. Update `wandering_merchant` weight from 18 to 15.
4. Update `trapped_chest` weight from 12 to 8.
5. Update `lore_fragment` weight from 15 to 12.
6. Keep `environmental_hazard` at 7.
7. Verify total weight distribution feels reasonable (~114 total).

**Files**: `js/eventSystem.js` (6 weight values changed)

**Validation**: `treasure_cache` + `trapped_chest` represent ~20% of total weight, down from ~42%.

### T002 — Anti-repetition logic

**Purpose**: Prevent the same event from firing twice consecutively, which feels bad.

**Steps**:
1. Track `lastEventId` (already exists as module variable).
2. In `pickEvent()`, halve the weight of `lastEventId` during selection (soft suppression, not hard block).
3. Additionally track `lastTwoEvents` — if an event appeared in both of the last two triggers, set its weight to 0 for this pick.
4. Reset tracking on floor change.

**Files**: `js/eventSystem.js` (~20 lines changed in `pickEvent()`)

**Validation**: Over 100 simulated picks, no event appears 3+ times consecutively. The same event can still appear after one different event in between.

### T003 — Depth-based probability and event unlock tuning

**Purpose**: Ensure event frequency and type availability scale well with dungeon depth.

**Steps**:
1. Keep existing probability formula: `chance = 0.35 + (depth - 1) * 0.02`, capped at 0.55.
2. Verify `minDepth` values for all 12 events create a good unlock progression:
   - Depth 1: treasure_cache, lore_fragment (safe, introductory)
   - Depth 2: ambush, trapped_chest, shrine_buff, gambling (mild risk)
   - Depth 3: wandering_merchant, npc_rescue, puzzle_room (interactive)
   - Depth 4: cursed_item (high risk/reward)
   - Depth 5: elite_ambush, environmental_hazard (dangerous)
3. Adjust any `minDepth` values that don't match this progression.

**Files**: `js/eventSystem.js` (minDepth adjustments)

**Validation**: At depth 1, only 2 events are eligible. By depth 5, all 12 are in the pool. The unlock cadence introduces ~2 new events per depth level.

## Definition of Done

- Chest events represent ~20% of the event pool (down from ~42%).
- No event fires 3+ times consecutively across normal play.
- Event unlock progression is gradual: 2 at depth 1, all 12 by depth 5.
- Probability formula unchanged (35%+2%/depth, cap 55%).
- No regressions in any event handler functionality.
- Manual playtest across 5 dungeon runs shows noticeable variety improvement.

## Risks

| Risk | Mitigation |
|---|---|
| Weight changes may make some events too rare | Log event distribution during playtest; adjust if needed |
| Anti-repetition could starve small-weight events | Use soft suppression (halve weight) not hard block |
| minDepth changes may lock out content too long | Keep most events available by depth 3 |
