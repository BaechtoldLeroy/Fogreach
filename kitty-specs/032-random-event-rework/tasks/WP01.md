# WP01 — Choice UI overlay + event architecture refactor

## Objective

Refactor `js/eventSystem.js` to support player-facing choices within events. Currently all events fire-and-forget — the handler runs instantly with no player input. This WP adds a generic choice dialog overlay (2-3 buttons) and restructures the event handler signature so events can present options and branch on the player's decision. Every downstream WP (shrine, NPC rescue, gambling, cursed items) depends on this choice infrastructure.

## Context

Current state (`js/eventSystem.js`):
- `EVENT_TYPES` is a flat array of `{id, name, weight, minDepth, handler(scene)}`.
- Handlers execute immediately — no async flow, no player input.
- `showEventToast()` displays a passive notification but has no interactive elements.

After this WP, events can optionally return a choice descriptor that triggers an overlay dialog.

## Subtasks

### T001 — Create `showEventChoiceDialog(scene, options)` function

**Purpose**: Render a modal overlay with 2-3 option buttons that the player can click/tap.

**Steps**:
1. Add `showEventChoiceDialog(scene, title, choices)` inside the eventSystem IIFE.
2. `choices` is an array of `{label, callback}` objects (max 3).
3. Create a dark semi-transparent overlay using `scene.add.rectangle()` covering the full viewport.
4. Render the title text centered, then one button per choice below it.
5. Each button dispatches its callback, destroys the overlay, and resumes the game.
6. While the overlay is visible, pause enemy movement (set `scene._eventChoiceActive = true`; enemies check this flag).

**Files**: `js/eventSystem.js` (~80 lines added)

**Validation**: Calling `showEventChoiceDialog(scene, 'Test', [{label:'A', callback:()=>console.log('A')}, {label:'B', callback:()=>console.log('B')}])` renders a modal with two buttons. Clicking one logs the value and destroys the overlay.

### T002 — Extend event handler signature to support choices

**Purpose**: Allow event handlers to return a choice descriptor instead of executing immediately.

**Steps**:
1. Change the event dispatch flow in `triggerEvent()`: if a handler returns an object `{title, choices}`, pass it to `showEventChoiceDialog()`.
2. If the handler returns `undefined`/`null` (existing behavior), proceed as before — fire-and-forget.
3. This is backward-compatible: all 6 existing events return nothing, so they continue working unchanged.

**Files**: `js/eventSystem.js` (~15 lines changed)

**Validation**: Existing events still fire normally. A test event returning `{title:'Pick', choices:[...]}` triggers the dialog.

### T003 — Add `registerEventType(eventDef)` public API

**Purpose**: Allow downstream WPs to register new event types without editing the hardcoded array.

**Steps**:
1. Export `window.EventSystem.registerEventType(def)` that pushes onto `EVENT_TYPES`.
2. Validate required fields (`id`, `name`, `weight`, `minDepth`, `handler`).
3. Reject duplicates by `id`.

**Files**: `js/eventSystem.js` (~20 lines added)

**Validation**: `EventSystem.registerEventType({id:'test', name:'Test', weight:10, minDepth:1, handler:()=>{}})` succeeds. Calling it again with the same `id` throws.

### T004 — Mobile-friendly touch targets for choice buttons

**Purpose**: Ensure choice buttons work on mobile.

**Steps**:
1. Choice buttons must be at least 44x44 CSS px (same standard as mobile controls).
2. Use `setInteractive()` with appropriately sized hit areas.
3. Space buttons with sufficient gap to prevent mis-taps.

**Files**: `js/eventSystem.js`

**Validation**: On mobile emulation, choice buttons are easily tappable without accidental overlap.

## Definition of Done

- `showEventChoiceDialog()` renders a working modal overlay with 2-3 buttons.
- Enemy movement pauses while choice is active.
- All 6 existing events work unchanged (backward compatible).
- `registerEventType()` is exported and validates input.
- Choice buttons are mobile-friendly (44x44 min).
- No console errors on desktop or mobile.

## Risks

| Risk | Mitigation |
|---|---|
| Overlay z-order conflicts with HUD elements | Use `setDepth(9999)` for the overlay layer |
| Enemy pause flag not checked everywhere | Audit enemy update loops; add `_eventChoiceActive` guard |
| Touch events may propagate through overlay | Call `event.stopPropagation()` on the overlay background |
