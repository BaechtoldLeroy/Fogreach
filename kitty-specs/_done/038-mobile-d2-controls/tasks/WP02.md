# WP02 — Tap-to-attack targeting

## Objective

Implement tap-to-attack on mobile: tapping an enemy sprite targets it and initiates an attack. The player should move into range if needed, then attack the targeted enemy.

## Context

WP01 establishes the tap detection infrastructure in `mobileControls.js`. This WP extends that system to detect taps on enemy sprites. The existing `mobileAutoAim.js` has nearest-enemy lookup utilities (from spec 021) that can be reused. Desktop D2-style attack targeting serves as the reference behavior.

## Subtasks

### T001 — Enemy tap detection via hit-testing

**Purpose**: Detect when the player taps on an enemy sprite rather than on the ground.

**Steps**:
1. In the tap handler from WP01, before processing as a move command, check if the tap world coordinates overlap an active enemy sprite.
2. Use Phaser's hit-testing (`scene.physics.overlapCirc` or bounding-box check against `window.enemies.children`) to find enemies under the tap point. Use a generous hit area (enemy bounds + ~16px padding) for fat-finger tolerance on mobile.
3. If an enemy is found, set it as the player's current target instead of triggering move-to.

**Files**: `js/mobileControls.js`, `js/mobileAutoAim.js`.

**Validation**: Tapping directly on an enemy highlights/targets it instead of walking to that position.

### T002 — Move-to-attack behavior

**Purpose**: When an enemy is targeted, the player moves into attack range and auto-attacks.

**Steps**:
1. When a target enemy is set, calculate distance to the enemy. If outside melee/attack range, set a movement target toward the enemy.
2. Each update frame, re-check distance. When within attack range, trigger `attack()` facing the enemy.
3. If the enemy moves, update the movement target to follow it.
4. Clear the target when the enemy dies, goes inactive, or the player taps elsewhere (ground or another enemy).

**Files**: `js/mobileControls.js`, `js/player.js`.

**Validation**: Tap an enemy across the room -- player walks to it and attacks on arrival. If enemy dies, player stops.

### T003 — Ranged ability integration

**Purpose**: For ranged abilities, tap-to-attack should fire toward the tapped enemy without requiring the player to walk into melee range.

**Steps**:
1. When a ranged ability is active and the player taps an enemy, set `lastMoveDirection` toward the enemy and fire the ability.
2. Reuse `mobileAutoAim.js` utilities (`findNearestEnemy`, direction computation) for consistency.
3. Respect the `autoAim` setting -- if disabled, ranged abilities fire in the current facing direction as before.

**Files**: `js/mobileAutoAim.js`.

**Validation**: With a ranged ability equipped, tapping a distant enemy fires toward it without walking. With autoAim off, tapping an enemy does not change aim direction.

## Definition of Done

- Tapping an enemy on mobile targets and attacks it.
- The player moves into range before attacking if too far away for melee.
- Ranged abilities fire toward the tapped enemy without requiring movement.
- Tapping the ground clears the enemy target and resumes normal tap-to-move.
- Desktop controls remain unaffected.

## Risks

- Fat-finger accuracy: enemy hit areas may need to be larger than sprite bounds on small screens.
- Following a moving enemy could cause stuttery movement if update frequency is too low.
