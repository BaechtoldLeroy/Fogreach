# WP01: Harden Walkability and Reachability Validation

## Branch Strategy

- **Planning base branch**: `main`
- **Merge target branch**: `main`
- **Implementation command**: `spec-kitty implement WP01`
- Note: The actual `base_branch` may differ for stacked WPs during `/spec-kitty.implement`

## Objective

Fix the core spawn validation pipeline so that enemies can only spawn on tiles that are genuinely walkable floor tiles AND reachable from the room's entry point. Currently, `isSpawnPositionBlocked` only checks for overlap with the `obstacles` physics group, which misses wall tiles and narrow gaps. The `recomputeAccessibleArea` flood-fill seeds from the player's current position, which can be unreliable if the player is briefly inside geometry.

## Context

This is a Phaser 3 browser-based action RPG. Procedural rooms are generated by `proceduralRooms.js` (BSP algorithm) which produces a tile grid where `'.'` = floor and `'#'` = wall. Room obstacles are created as Phaser static physics bodies in `roomManager.js`. The accessible-area system in `roomManager.js` (lines 1080-1270) performs a flood-fill to find reachable cells, then builds a `spawnCandidates` pool used by `pickAccessibleSpawnPoint`.

**Key files**:
- `js/roomManager.js` -- `isSpawnPositionBlocked()` (line 1047), `recomputeAccessibleArea()` (line 1080), `pickAccessibleSpawnPoint` (line 1238)
- `js/proceduralRooms.js` -- BSP room generation, `floodAccessible()` (line 551), doorway data, wall grid

**Constants** (roomManager.js):
- `ROOM_SPAWN_HALF_SIZE = 8` -- half-size for spawn candidate bounding box
- `ACCESS_GRID_SIZE = 32` -- cell size for the accessible-area grid
- `MIN_PLAYER_SPAWN_DISTANCE = 160` -- minimum distance from player for spawn candidates

## Subtask T001: Harden `isSpawnPositionBlocked` to Check Walkable Tiles

**Purpose**: The current `isSpawnPositionBlocked` (line 1047) only checks overlap against the `obstacles` physics group. This means a point between two walls, in a narrow 1-tile gap, or in an isolated pocket with no physics obstacle body may pass the check. We need to also validate against the procedural room's tile grid.

**Steps**:

1. **Expose the room wall grid to `roomManager.js`**:
   - In `proceduralRooms.js`, the BSP generator returns a `walls` array (array of strings, `'#'` = wall, `'.'` = floor). This is already returned in the room data object (line 527: `walls: walls`).
   - In `roomManager.js`, when `enterRoom()` sets up a room, the wall grid should be stored on the scene or in a module-level variable so `isSpawnPositionBlocked` can reference it.
   - Check if `enterRoom()` already stores room data -- look for where `room.walls` or `room.grid` is used. If not stored, add: `scene._currentRoomWalls = room.walls;` and also store `scene._currentRoomTileSize` and `scene._currentRoomBounds` for coordinate mapping.

2. **Add tile-grid check to `isSpawnPositionBlocked`**:
   - After the existing obstacle-overlap check, add a wall-tile check:
     ```javascript
     // Check against procedural room wall grid
     const walls = scene?._currentRoomWalls;
     if (walls && walls.length > 0) {
       const tileSize = scene._currentRoomTileSize || 48; // match proceduralRooms tile size
       const roomBounds = scene._currentRoomBounds;
       const tileX = Math.floor((px - roomBounds.x) / tileSize);
       const tileY = Math.floor((py - roomBounds.y) / tileSize);
       if (tileY < 0 || tileY >= walls.length || tileX < 0 || tileX >= walls[0].length) {
         return true; // out of grid = blocked
       }
       if (walls[tileY][tileX] !== '.') {
         return true; // wall tile = blocked
       }
     }
     ```
   - The scene reference needs to be accessible -- `isSpawnPositionBlocked` currently uses the global `obstacles` variable. Add `scene` as an optional parameter or use the obstacles group's scene reference: `obstacles?.scene`.

3. **Determine the tile size mapping**:
   - In `proceduralRooms.js`, find where the tile grid is converted to pixel coordinates. Look for the tile size constant (likely 48px based on standard Phaser tile sizes, but check `enterRoom()` for the actual scale factor used when placing wall sprites/obstacles).
   - The coordinate mapping must match: `pixelX = roomBounds.x + tileX * tileSize`.

4. **Handle edge case: non-procedural rooms**:
   - Template-based rooms (from `roomTemplates.js`) may not have a `walls` grid. The function should gracefully skip the tile check when `_currentRoomWalls` is null, falling back to obstacle-only checking.

**Files**: `js/roomManager.js`, `js/proceduralRooms.js`

**Validation**:
- [ ] `isSpawnPositionBlocked` returns `true` for coordinates on wall tiles
- [ ] `isSpawnPositionBlocked` returns `true` for coordinates outside the grid bounds
- [ ] `isSpawnPositionBlocked` still works correctly for template rooms (no wall grid)
- [ ] No console errors when wall grid is not available

## Subtask T002: Ensure Flood-Fill Seeds from Room Entry/Doorway

**Purpose**: `recomputeAccessibleArea` (line 1080) starts the flood-fill from the player's current position. If the player is briefly overlapping geometry (e.g., during a room transition animation), the seed point may be blocked, causing the entire accessible area to be null -- which means `pickAccessibleSpawnPoint` returns null for all candidates. The flood-fill should seed from the room's entry doorway position, which is guaranteed to be on a floor tile.

**Steps**:

1. **Store doorway/entry position on the scene**:
   - In `enterRoom()` (roomManager.js), when the player enters a new room, store the entry position:
     ```javascript
     scene._roomEntryPoint = { x: playerSpawnX, y: playerSpawnY };
     ```
   - The player spawn position in `enterRoom` is derived from the room's `playerSpawn` data (from `proceduralRooms.js` line 381), which is the center of the first accessible chamber -- this is a reliable floor position.

2. **Modify `recomputeAccessibleArea` to prefer entry point as seed**:
   - At line 1136, where `startCX`/`startCY` are computed from `activePlayer.x`/`activePlayer.y`, add a preference for `scene._roomEntryPoint`:
     ```javascript
     const seedX = scene._roomEntryPoint?.x ?? activePlayer.x;
     const seedY = scene._roomEntryPoint?.y ?? activePlayer.y;
     let startCX = Phaser.Math.Clamp(Math.floor((seedX - bounds.x) / cellSize), 0, cols - 1);
     let startCY = Phaser.Math.Clamp(Math.floor((seedY - bounds.y) / cellSize), 0, rows - 1);
     ```
   - Keep the existing fallback spiral search (lines 1139-1158) in case the entry point is somehow blocked.

3. **Verify timing of `recomputeAccessibleArea` call**:
   - It is called at line 402 in `enterRoom()`. Verify this happens AFTER obstacles are placed and the player position is set, so the grid accurately reflects the room layout.
   - Also called in `wave.js` line 178 after `placeObstaclesForWave` -- this is correct (recompute after obstacles change).

4. **Handle the case where `_roomEntryPoint` is not set**:
   - For the first room or if the entry point is missing, fall back to player position (existing behavior).

**Files**: `js/roomManager.js`

**Validation**:
- [ ] Flood-fill seeds from room entry point, not arbitrary player position
- [ ] Accessible area is correctly computed even if player sprite overlaps geometry briefly
- [ ] `recomputeAccessibleArea` still works when `_roomEntryPoint` is not set (fallback to player)
- [ ] Timing: `recomputeAccessibleArea` is called after obstacles are placed

## Subtask T003: Filter Spawn Candidates with Wall-Adjacency Buffer

**Purpose**: Even with correct flood-fill reachability, enemies spawned in cells directly adjacent to walls can clip into geometry or appear stuck. Add a buffer that excludes cells neighboring wall tiles from the `spawnCandidates` pool (keep them in `fallbackCandidates` for dire situations).

**Steps**:

1. **Locate where `spawnCandidates` are built** in `recomputeAccessibleArea`:
   - After the flood-fill BFS (line 1175+), reachable cells are added to either `spawnCandidates` or `fallbackCandidates` based on distance from the player.
   - Find the exact condition that distinguishes them.

2. **Add wall-adjacency check**:
   - For each reachable cell `(cx, cy)`, check if any of its 8 neighbors (including diagonals) is blocked:
     ```javascript
     const isAdjacentToWall = (cx, cy) => {
       for (let dx = -1; dx <= 1; dx++) {
         for (let dy = -1; dy <= 1; dy++) {
           if (dx === 0 && dy === 0) continue;
           if (isBlocked(cx + dx, cy + dy)) return true;
         }
       }
       return false;
     };
     ```
   - If `isAdjacentToWall` is true, add to `fallbackCandidates` only (not `spawnCandidates`).
   - This keeps wall-adjacent cells available as a last resort but prefers interior cells.

3. **Ensure `spawnCandidates` is not empty after filtering**:
   - In very small rooms, all cells might be adjacent to walls. If `spawnCandidates` is empty after filtering, `pickAccessibleSpawnPoint` already falls back to `fallbackCandidates` (line 1241-1243), so this is safe.

4. **Performance consideration**:
   - The adjacency check calls `isBlocked` which uses a cache (`blockCache`). Since the flood-fill already populated this cache for visited cells, the lookup is O(1). The 8-neighbor check per cell is negligible.
